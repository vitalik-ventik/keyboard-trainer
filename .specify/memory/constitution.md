# Keyboard Trainer Constitution

Клавіатурний тренажер у стилі Geometry Dash: модульний веб-застосунок на чистому
HTML5 Canvas + Vanilla JS (ES6 Modules) з українською розкладкою ЙЦУКЕН.

## Core Principles

### I. Чистий стек: HTML5 Canvas + Vanilla JS (NON-NEGOTIABLE)

Проект MUST залишатися у межах чистого HTML5 (Canvas 2D Context) + Vanilla JS
(ES6 Modules). Заборонено пропонувати або впроваджувати Node.js, Webpack, React,
Vue, npm-пакети чи будь-які інші фреймворки та системи збірки. Порушення цього
принципу вважається критичною помилкою.

**Обґрунтування**: Застосунок має запускатися локально у браузері без етапу
збірки; будь-яка зовнішня залежність руйнує цю гарантію та ускладнює супровід.

### II. Модульна архітектура файлів

Код MUST розділятися на `index.html`, `css/style.css` та окремі JavaScript-файли
в папці `js/` (`main.js`, `assets.js`, `engine.js`, `keyboard.js`), пов'язані
через ES6-імпорти або правильний порядок підключення. Нові модулі MUST
створюватися як окремі файли в `js/` з єдиною відповідальністю. Мова інтерфейсу
та коментарів — виключно українська.

**Обґрунтування**: Чітке розділення на логічні файли забезпечує читабельність,
незалежне тестування модулів і контрольовану еволюцію архітектури без
інструментів збірки.

### III. Canvas-рендеринг та процедурна графіка

Усі ігрові об'єкти (кубик, шипи), візуальна клавіатура, меню та фони MUST
рендеритися на HTML5 Canvas. Жодного DOM-маніпулювання для ігрових об'єктів.
Бекграунди MUST генеруватися процедурно (паралакс-сітки, геометричні гори,
пульсуючі еквалайзери для Бос-рівня) — статичні зображення для фонів заборонені.
Стилістика — темний неоновий кіберпанк (Geometry Dash) з адаптивністю під
роздільну здатність екрана Windows 11.

**Обґрунтування**: Canvas гарантує стабільні 60 FPS для ігрового циклу;
процедурна графіка усуває залежність від бінарних асетів та дозволяє
синхронізацію візуалу з музикою.

### IV. Локальні ресурси та стійкість до помилок

Звук та музика MUST завантажуватися асинхронно виключно з локальних файлів
(`sounds/*.wav`, `music/*.mp3`). Категорично заборонено аудіо-посилання на
зовнішні CDN чи хостинги. Збереження прогресу — виключно локальний
`localStorage`. Кожне звернення до `localStorage` та кожне завантаження аудіо
MUST бути обгорнуте у `try/catch` для запобігання фатальних помилок у консолі
Windows 11.

**Обґрунтування**: Повна автономність застосунку (офлайн-робота) та захист від
збоїв середовища (заборонений storage, відсутній файл, автоплей-політики
браузера).

### V. Ігрові стани та механіка введення ЙЦУКЕН

Ігрові стани MUST бути чітко розділені: `MENU`, `SETTINGS`, `LEVEL_SELECT`,
`PLAYING`, `GAMEOVER`, `VICTORY`; музичні треки MUST перемикатися відповідно до
активного стану. Ігровий цикл MUST працювати через `requestAnimationFrame`.
Введення: тільки українська розкладка ЙЦУКЕН; обробник `keydown` MUST
скасовувати дефолтну поведінку Windows (`preventDefault` для Пробілу та
Стрілок). Візуальна клавіатура рендериться виключно на Canvas і відображає лише
блоки з українськими літерами — службові клавіші (Shift, Enter, CapsLock) NOT
відображаються та NOT прораховуються. Режими складності MUST реалізовувати:
`EASY` — вибух лише при фізичному зіткненні з шипом (помилкові натискання
завчасно ігноруються); `HARD` — будь-яка помилка, пропуск таймінгу чи
неправильна літера — миттєвий вибух.

**Обґрунтування**: Це ядро геймплею; розмиття меж станів або механіки введення
робить поведінку гри непередбачуваною і нетестованою.

### VI. Повнота коду без скорочень

Заборонено використовувати коментарі-заглушки на кшталт `// тут додайте логіку`
чи `// решта коду без змін`. Код кожної функції або класу в межах таски MUST
бути написаний повністю від початку до кінця. ES6+ код MUST бути чистим і
оптимізованим, без сторонніх npm-пакетів.

**Обґрунтування**: Часткові фрагменти коду ламають робочий застосунок без
системи збірки та ускладнюють перевірку відповідності конституції.

## Технічні обмеження (Tech Stack & Constraints)

- **Мова**: українська — для інтерфейсу, коментарів та документації.
- **Формат проекту**: модульний Vanilla JS застосунок без етапу збірки;
  запуск — відкриття `index.html` (або локальний статичний сервер для
  ES6-модулів).
- **Графіка**: HTML5 Canvas (2D Context); DOM використовується лише для
  контейнера Canvas та допоміжної стилізації через `css/style.css`.
- **Аудіо**: локальні `sounds/*.wav` та `music/*.mp3`; зовнішні URL заборонені.
- **Персистентність**: лише `localStorage`; жодних мережевих запитів для
  збереження прогресу.
- **Цільова платформа**: сучасні браузери на Windows 11; продуктивність —
  стабільні 60 FPS ігрового циклу.

## Робочий процес та ролі ШІ-агентів

Будь-який ШІ-агент, що працює з репозиторієм, MUST приймати одну з ролей
залежно від запиту:

- **[Lead Architect]**: контроль цілісності модульної архітектури; блокує
  будь-які пропозиції переходу на фреймворки чи системи збірки (Принцип I).
- **[UI/UX Engineer]**: рендеринг меню, модальних вікон та візуальної
  клавіатури на Canvas; дотримання стилістики темного неонового кіберпанку та
  адаптивності під Windows 11 (Принцип III).
- **[Core Developer]**: ігровий цикл (`requestAnimationFrame`), колізії,
  State Manager, обробка `keydown`; чистий ES6+ без npm-пакетів
  (Принципи V, VI).

Перевірка якості: кожна зміна перед завершенням MUST проходити ручну перевірку
на відповідність усім шести принципам (Constitution Check у plan.md). Порушення
будь-якого пункту Технічних обмежень вважається критичною помилкою і MUST бути
виправлене до злиття змін.

## Governance (Управління)

Ця конституція має пріоритет над усіма іншими практиками та інструкціями в
репозиторії. У разі конфлікту з іншими документами (включно з AGENTS.md) діє
конституція; конфлікт MUST бути усунений синхронізацією документів.

**Процедура внесення змін**: будь-яка поправка MUST бути задокументована через
оновлення цього файлу з Sync Impact Report, обґрунтуванням та новою версією.
Зміни, що зачіпають Принципи I–VI, MUST супроводжуватися перевіркою залежних
шаблонів (`.specify/templates/*.md`) та AGENTS.md.

**Версіонування** (семантичне):

- **MAJOR** — видалення або несумісне перевизначення принципу чи розділу
  управління.
- **MINOR** — додавання нового принципу/розділу або суттєве розширення вимог.
- **PATCH** — уточнення формулювань, виправлення помилок без зміни семантики.

**Перевірка відповідності**: усі плани (`/speckit.plan`) MUST проходити
Constitution Check перед фазою досліджень і повторно після дизайну. Будь-яке
ускладнення, що виходить за межі Принципів I–II, MUST бути обґрунтоване у
таблиці Complexity Tracking відповідного plan.md.
