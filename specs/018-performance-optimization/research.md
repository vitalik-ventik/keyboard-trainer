# Research: Оптимізація продуктивності для слабких ноутбуків

**Feature**: 018-performance-optimization
**Date**: 2026-07-20

---

## R1: Off-screen Canvas кешування процедурних фонів

### Decision

Використовувати окремий `HTMLCanvasElement` (невидимий, не доданий у DOM) для рендеру фону. Фон перемальовується на off-screen canvas раз на 3 кадри (кожні ~50ms при 60 FPS), пропускаючи 2 проміжні кадри. Основний canvas композитує фон через `ctx.drawImage(bgCanvas, 0, 0)` кожен кадр — одна операція замість десятків.

### Rationale

- Усі 31 тема фону разом виконують сотні Canvas-викликів на кадр (fillRect, stroke, arc, createLinearGradient, shadowBlur, fillText). На Intel UHD це займає 5-12ms з 16.6ms бюджету — залишаючи <5ms на решту рендеру.
- `drawImage` з off-screen canvas — одна з найдешевших операцій Canvas2D (<0.1ms), оскільки GPU вже має текстуру в пам'яті.
- Оновлення раз на 3 кадри зберігає плавність анімації (60→20 fps для фону) — непомітно для людського ока на тлі, але вивільняє 60%+ бюджету рендеру.
- Для тем із швидкою анімацією (inferno_core, equalizer) оновлення можна пришвидшити до кожного кадру — але навіть там більшість елементів (тріщини, іскри) можна рендерити поверх закешованого статичного тла.

### Alternatives Considered

| Альтернатива | Чому відхилено |
|-------------|----------------|
| Переписування фонів на WebGL | Порушує Constitution I (Vanilla JS + Canvas2D), надскладна міграція 31 теми |
| Зменшення складності фонів (видалення ефектів) | Порушує SC-006 (візуальна ідентичність), гра втрачає стиль |
| CSS-анімації замість Canvas фону | Порушує Constitution III (Canvas-only рендер) |
| Кешування кожного кадру (60 fps кеш) | Безглуздо — немає виграшу, лише додатковий drawImage |

## R2: Заміна ctx.shadowBlur на градієнтні ореоли

### Decision

Усі виклики `ctx.shadowBlur` зі значенням > 4px замінити на один із трьох патернів залежно від контексту:
1. **Для прямокутних/квадратних об'єктів (скіни гравця, клавіші)**: малювати градієнтний ореол через `createLinearGradient` або `createRadialGradient` навколо об'єкта перед малюванням самого об'єкта.
2. **Для кіл/дуг (перешкоди saw, орбіти)**: використовувати `createRadialGradient` з центром у об'єкті, радіусом = shadowBlur, із затуханням до transparent.
3. **Для тексту (матриця, еквалайзер)**: використовувати `fillText` із напівпрозорим кольором + зсув на 1-2px замість glow.

### Rationale

- `ctx.shadowBlur` змушує браузер обчислювати Gaussian blur для кожного пікселя фігури **на CPU** перед передачею на GPU. При shadowBlur > 10px це експоненційно дорожче.
- Градієнти (`createLinearGradient`, `createRadialGradient`) обчислюються один раз при створенні, а потім просто застосовуються як fillStyle — GPU-native операція.
- Візуальний результат ідентичний: неоновий glow виглядає як градієнт від яскравого центру до прозорого краю — що і є фізичною природою світіння.

### Implementation Map

| Де використовується | Тип заміни |
|--------------------|-----------|
| Скіни гравця (SKIN_RENDERERS, 31 функція) | RadialGradient ореол + зменшений shadowBlur (2-4px) |
| drawSpike / drawDoubleSpike / drawSaw | RadialGradient підкладка під шип |
| drawKeyboard (glow for target/error/group) | RadialGradient + напівпрозорий stroke |
| Background themes (demon, inferno_core, aurora_wings, etc.) | RadialGradient замість applyGlow → shadowBlur |
| renderHitWindow (зонний індикатор) | Лінійний градієнт замість glow |

### Alternatives Considered

| Альтернатива | Чому відхилено |
|-------------|----------------|
| `ctx.filter = 'blur(4px)'` замість shadowBlur | filter blur теж виконується на CPU і може бути навіть повільнішим за shadowBlur у Chrome |
| Зменшення всіх shadowBlur до ≤ 4px | Не дає достатнього приросту, і на 4px glow майже не видно |
| Видалення glow зовсім | Порушує SC-006, втрата стилю неонового кіберпанку |

## R3: Оптимізація частинкових систем

### Decision

Застосувати **Object Pooling** для всіх частинкових систем (фон, вибухи, perfect-ефекти). Пул має фіксований розмір (max 30). При перевищенні ліміту найстаріша (з найменшим life) частинка перевикористовується замість створення нової. Використати єдиний `ParticlePool` у `cache.js`.

### Rationale

- Поточний код у `backgrounds.js` вже частково реалізує pooling (рядок 9: пошук частинки з min life, `splice`). Але кожна тема має власний пул і ліміти різні (MAX_PARTICLES=50, MAX_RAINDROPS=120, MAX_MATRIX_COLUMNS=40).
- Створення нових об'єктів через `{}` і `push()` у гарячому циклі викликає GC-паузи, що помітні як мікро-фризи на слабких пристроях.
- Об'єднання всіх пулів в один уніфікований `ParticlePool`:
  - Виділяє 30 об'єктів один раз при створенні
  - `acquire()` повертає вільну частинку або перезаписує найстарішу
  - `release()` позначає як вільну (без splice!)
  - Жодних алокацій у гарячому циклі

### Alternatives Considered

| Альтернатива | Чому відхилено |
|-------------|----------------|
| Просто зменшити ліміти (без pooling) | Не вирішує GC-проблему, старі частинки все ще створюються через `{}` |
| Web Workers для фізики частинок | Порушує модульну архітектуру, надмірне ускладнення |
| Видалення частинкових систем | Порушує SC-006, візуальна ідентичність |

## R4: Управління частотою кадрів (Frame Controller)

### Decision

Додати в `main.js` логіку пропуску кадру: якщо `dt > 0.03` секунди (33ms, тобто <30 FPS), пропустити `update()` і `render()` для цього кадру — лише викликати наступний `requestAnimationFrame`. Це запобігає накопиченню відставання (спіраль смерті), коли гра намагається надолужити пропущений час і ще більше сповільнюється.

Додатково: кламп `dt` залишити на `Math.min(dt, 0.05)` для нормальних кадрів, але додати нижню межу пропуску.

### Rationale

- При великому dt (після розблокування вкладки, після GC-паузи) фізика «стрибає» вперед, потенційно пропускаючи колізії або викликаючи миттєвий game over без видимої причини.
- Пропуск кадру дає браузеру час «віддихатися» і повернутися до стабільних 60 FPS швидше, ніж спроба наздогнати.

### Alternatives Considered

| Альтернатива | Чому відхилено |
|-------------|----------------|
| Фіксований крок фізики (fixed timestep) | Вимагає переписування всього update-циклу та інтерполяції рендеру між станами — надмірно для цього проекту |
| Просто зменшити кламп до 0.03 | Недостатньо — при різкому стрибку фізика все одно стрибне занадто далеко |
| requestAnimationFrame з власним таймером на 60fps | Ускладнює код без значного виграшу, rAF вже синхронізований з vsync |

## R5: Аудіо-оптимізація (victory.wav → MP3)

### Decision

Конвертувати `sounds/victory.wav` (4.3MB, PCM 16-bit, 44100Hz stereo) у `sounds/victory.mp3` із бітрейтом 192kbps. Файл `victory.wav` видалити. Оновити `SOUND_FILES` у `assets.js` — замінити `"victory.wav"` на `"victory.mp3"`. Завантаження залишається асинхронним через `fetch()` + `decodeAudioData()`.

### Rationale

- 4.3MB WAV — це 90% загального розміру звукових асетів (4.78MB). Декодування PCM у AudioBuffer на слабкому CPU займає 200-500ms і блокує головний потік.
- MP3 192kbps для 11-секундного кліпу = ~260KB. Декодування в 15-20 разів швидше.
- Інші sound-файли (jump.wav 13KB, click.wav 176KB, explode.wav 282KB) не потребують конвертації — їхній сумарний внесок у затримку незначний.
- Музика (music/*.mp3) уже в MP3 — оптимізації не потребує.

### Alternatives Considered

| Альтернатива | Чому відхилено |
|-------------|----------------|
| OGG Vorbis | Гірша підтримка в Safari/Edge, більший розмір при тому ж бітрейті |
| AAC (.m4a) | Дорожче декодування на слабких CPU |
| Opus (.webm) | Найкращий кодек, але `decodeAudioData` не підтримує Opus у Firefox без додаткових флагів |
| Стримінг аудіо (MediaElement) для victory | Декодування все одно відбувається, просто в іншому потоці — не вирішує проблему затримки відтворення |

## R6: Кешування градієнтів

### Decision

Усі градієнти (`createLinearGradient`, `createRadialGradient`), що використовуються в render-функціях фонових тем, створювати один раз при виклику `BackgroundRenderer.init(W, H, groundY)` (тобто при старті рівня або ресайзі) і зберігати в об'єкті `_gradientCache`. При рендері перевикористовувати закешовані градієнти через зміну `ctx.fillStyle = _gradientCache.themeX_gradientY`.

### Rationale

- Поточний код створює градієнти в тілі render-функції — тобто кожен кадр. Наприклад, `renderParallaxCity` створює 3 градієнти щокадру. `renderInfernoCore` — 5+ градієнтів. На 31 темі це сотні алокацій градієнтів на кадр.
- Canvas-градієнт після створення є незмінним об'єктом, прив'язаним до координат. Якщо координати не змінюються (розмір полотна стабільний), градієнт можна перевикористовувати без перестворення.
- Для градієнтів, що залежать від `time` або `speed`, кешування не застосовується — вони мають оновлюватися покадрово (але таких меншість).

### Alternatives Considered

| Альтернатива | Чому відхилено |
|-------------|----------------|
| Не кешувати — просто зменшити кількість градієнтів | Градієнти — це суть візуального стилю (неонові переходи, затухання). Видалення погіршить вигляд. |
| Перейти на CSS linear-gradient для Canvas | Неможливо — Canvas не підтримує CSS-градієнти безпосередньо |
