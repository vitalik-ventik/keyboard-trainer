# Feature Specification: Оптимізація продуктивності для слабких ноутбуків

**Feature Branch**: `018-performance-optimization`

**Created**: 2026-07-20

**Status**: Draft

**Input**: User description: "Оптимізація продуктивності гри на слабких ноутбуках. На пристроях з інтегрованою графікою (Intel UHD, AMD Vega) спостерігається просідання FPS нижче 30 при активній грі через: (1) процедурні фони зі складними ефектами (градієнти, glow, частинки, синусоїди) що перемальовуються кожен кадр; (2) використання ctx.shadowBlur (10-60px) у рендері скінів, перешкод та фонів — найповільнішої операції Canvas2D; (3) повний рендер візуальної клавіатури (41 клавіша з roundRect, glow, текстом) кожен кадр; (4) відсутність кешування статичних/повільнозмінних елементів через off-screen canvas; (5) частинкові системи з надмірною кількістю (до 120 частинок на кадр); (6) великий файл victory.wav (4.3MB) що блокує аудіо-декодування."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Стабільна частота кадрів під час гри (Priority: P1)

Гравець запускає рівень на слабкому ноутбуці з інтегрованою графікою (Intel UHD, AMD Vega). Усі ігрові елементи — процедурний фон, кубик, перешкоди, клавіатура, частинки — рендеряться без затримок. Частота кадрів залишається у межах 50-60 FPS навіть під час інтенсивних рівнів (Бос-рівень, рівні з темами inferno_core, aurora_wings). Візуальний стиль неонового кіберпанку збережено без змін.

**Why this priority**: Це основна мета оптимізації — гра повинна бути доступною на широкому спектрі пристроїв. Без цього решта оптимізацій не має сенсу.

**Independent Test**: Запустити рівень 31 (ФІНАЛЬНИЙ ДЕМОН) на ноутбуці з Intel UHD Graphics і виміряти FPS до та після оптимізації. Після впровадження FPS має бути не нижче 50 кадрів/с.

**Acceptance Scenarios**:

1. **Given** ноутбук з Intel UHD 620, бос-рівень (рівень 31, тема inferno_core), **When** активний ігровий процес із рухом перешкод та стрибками, **Then** частота кадрів не падає нижче 50 FPS
2. **Given** ноутбук з AMD Vega 3, рівень 18 (тема rooftop_grid), **When** увімкнено демо-режим у головному меню, **Then** частота кадрів стабільна на 60 FPS
3. **Given** слабкий ноутбук, будь-який рівень, **When** відбувається вибух кубика з частинками, **Then** не спостерігається мікро-фризів чи стрибків кадру

---

### User Story 2 - Швидке завантаження аудіо-асетів без блокування (Priority: P2)

Гравець відкриває застосунок. Усі аудіо-файли (звуки та музика) завантажуються та декодуються асинхронно без помітних затримок. Великий файл victory.wav більше не спричиняє тривалого блокування при декодуванні в AudioContext. Індикатор завантаження показує прогрес без зависань.

**Why this priority**: 4.3MB WAV-файл спричиняє відчутну затримку на старті, особливо на слабких пристроях із повільними дисками. Це друга за важливістю проблема для UX.

**Independent Test**: Відкрити застосунок на слабкому ноутбуці. Екран завантаження має завершитися протягом 2 секунд, без блокування UI-потоку.

**Acceptance Scenarios**:

1. **Given** слабкий ноутбук із HDD, **When** перше завантаження застосунку, **Then** екран завантаження завершується протягом 3 секунд
2. **Given** будь-який пристрій, **When** рівень завершено перемогою, **Then** звук victory відтворюється без затримки
3. **Given** помилка завантаження аудіо-файлу, **When** файл відсутній або пошкоджений, **Then** гра продовжує роботу без аудіо, не викликаючи фатальної помилки в консолі

---

### User Story 3 - Візуальна клавіатура без затримок введення (Priority: P2)

Гравець натискає літери на фізичній клавіатурі. Візуальна клавіатура на Canvas миттєво оновлює підсвітку цільової літери, не спричиняючи затримок уведення. Під час швидкого набору (рівні з високою швидкістю) підсвітка клавіш не відстає від ігрового процесу.

**Why this priority**: Клавіатура — основний інтерфейс гри. Її повільний рендер безпосередньо впливає на геймплей, особливо на швидких рівнях (boss, master).

**Independent Test**: Запустити рівень 31 (speed multiplier x1.25 FAST). Візуальна підсвітка цільової літери має оновлюватися без видимого відставання від руху перешкод.

**Acceptance Scenarios**:

1. **Given** активний рівень зі швидкістю 450+ px/s, **When** гравець натискає правильну літеру, **Then** підсвітка клавіші змінюється протягом одного кадру (16ms)
2. **Given** активний рівень, **When** не відбувається змін у стані клавіатури (targetLetter, groupLetters, wrongKeyError), **Then** клавіатура не перемальовується (використовується кеш)
3. **Given** гравець натиснув помилкову літеру, **When** wrongKeyError активний (350 мс), **Then** червона підсвітка клавіші відображається без впливу на FPS

---

### User Story 4 - Плавний фоновий рендер без просідань (Priority: P3)

Гравець проходить різні рівні з різними темами фону (cyber_grid, starfield, aurora_wings, inferno_core тощо). Усі 31 тема рендеряться плавно без стрибків FPS. Частинкові ефекти (сніг, дощ, зірки, бульбашки, матриця) не перевантажують GPU. Градієнти та glow-ефекти виглядають ідентично до оптимізації, але рендеряться швидше.

**Why this priority**: Фони — найважча частина рендеру. Оптимізація фону дасть найбільший приріст FPS на слабких GPU, але це технічно складніша задача, ніж кешування клавіатури чи стиснення аудіо.

**Independent Test**: Порівняти скріншоти фонів до та після оптимізації для всіх 31 теми. Вони мають бути візуально ідентичними. FPS при цьому має зрости.

**Acceptance Scenarios**:

1. **Given** рівень із темою inferno_core (бос), **When** активний рендер із серцебиттям, тріщинами, іскрами, **Then** FPS не падає нижче 50
2. **Given** рівень із темою matrix_flow, **When** рендериться 60 колонок символів катакани, **Then** оновлення позицій колонок відбувається раз на 3 кадри без візуально помітної різниці
3. **Given** будь-яка тема фону, **When** використовується ctx.shadowBlur з параметром > 8px, **Then** цей виклик замінено на градієнтний ореол, візуальний результат ідентичний
4. **Given** тема з частинками (neon_rain, deep_abyss, toxic_waste), **When** кількість активних частинок досягає ліміту, **Then** нові частинки заміщують найстаріші без перевищення ліміту в 30 одиниць

---

### Edge Cases

- Що відбувається при швидкому ресайзі вікна (Windows 11 snap layouts)? Off-screen canvases мають перестворюватися без витоку пам'яті.
- Як поводиться кеш клавіатури при зміні роздільної здатності? Кеш має інвалідуватися при зміні розміру області клавіатури.
- Що станеться, якщо аудіо-файл відсутній або пошкоджений? Система має продовжити роботу без фатальної помилки (try/catch у всіх точках завантаження аудіо).
- Як поводиться гра при різкому стрибку dt (>100ms, наприклад після розблокування вкладки)? Кадр має бути пропущений без оновлення фізики (`if (dt > 0.03) return`).
- Що при переході між рівнями в демо-режимі? Кеш фону має очищатися, щоб уникнути показу фону попереднього рівня.
- Як перевірити візуальну ідентичність оптимізованих елементів? Усі glow-ефекти, кольори, прозорості та розміри мають співпадати з еталоном (скріншоти до/після).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST використовувати off-screen canvas для кешування процедурного фону. Фон MUST оновлюватися не частіше ніж раз на 3 кадри (кожні ~50ms при 60 FPS).
- **FR-002**: Система MUST використовувати off-screen canvas для кешування візуальної клавіатури. Кеш MUST інвалідуватися лише при зміні targetLetter, groupLetters або wrongKeyError.
- **FR-003**: Усі виклики `ctx.shadowBlur` зі значенням > 4px MUST бути замінені на градієнтні ореоли (radialGradient / linearGradient) із візуально ідентичним результатом.
- **FR-004**: Кількість активних частинок у будь-якій частинковій системі (фон, вибухи, perfect-ефекти) MUST не перевищувати 30 одночасно.
- **FR-005**: Градієнти (`createLinearGradient`, `createRadialGradient`) MUST створюватися один раз при ініціалізації теми або при зміні розміру полотна, а не кожен кадр.
- **FR-006**: Ігровий цикл MUST пропускати кадр (пропускати update + render), якщо `dt > 0.03` секунди, щоб уникнути накопичення відставання.
- **FR-007**: Файл `sounds/victory.wav` MUST бути конвертований у формат MP3 (або інший формат зі стисненням без втрати якості) з розміром не більше 500KB.
- **FR-008**: Усі точки завантаження аудіо (`loadAssets`, `playSound`, `playMusic`) MUST мати обгортку `try/catch` для запобігання фатальним помилкам.
- **FR-009**: Off-screen canvases MUST перестворюватися при зміні розміру вікна без витоку пам'яті (старі canvases мають бути видалені).
- **FR-010**: Візуальний вигляд усіх елементів (фони, скіни, перешкоди, клавіатура, частинки) MUST бути ідентичним до оптимізації — не допускається зміна кольорів, прозорості, розмірів або позицій.
- **FR-011**: Частота кадрів MUST залишатися в діапазоні 50-60 FPS на цільових пристроях (Intel UHD 620, AMD Vega 3) для всіх 31 рівня та всіх тем фону.
- **FR-012**: Усі зміни MUST дотримуватися Принципів I–VI Конституції проекту (Vanilla JS, Canvas-only рендер, процедурні фони, ES6 модулі, локальне збереження, повнота коду).

### Key Entities

- **Off-screen Canvas (BackgroundCache)**: невидимий canvas для кешування процедурного фону теми рівня. Містить посилання на 2D-контекст, поточну тему (bgTheme), лічильник кадрів для оновлення раз на N кадрів.
- **Off-screen Canvas (KeyboardCache)**: невидимий canvas для кешування візуальної клавіатури. Інвалідується при зміні targetLetter, groupLetters або wrongKeyError.
- **ParticlePool**: пул частинок з фіксованим лімітом (max 30). Старі частинки перевикористовуються замість створення нових (object pooling).
- **GradientCache**: колекція попередньо створених градієнтів для кожної теми фону. Перестворюється лише при зміні розміру полотна.
- **AudioAsset**: аудіо-файл після асинхронного завантаження та декодування. Містить AudioBuffer для звукових ефектів та MediaElement для музики.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Усі 31 рівень досягають стабільних 50-60 FPS на ноутбуці з Intel UHD 620 при роздільній здатності 1366x768.
- **SC-002**: Час завантаження аудіо-асетів (від відкриття сторінки до завершення loading screen) не перевищує 3 секунди.
- **SC-003**: Візуальна клавіатура оновлює підсвітку протягом одного кадру (≤16ms) — без помітного відставання.
- **SC-004**: Кількість викликів `ctx.shadowBlur` у render-циклі зменшена до 0 (нуль) для всіх тем фону та скінів.
- **SC-005**: Розмір завантажуваних аудіо-даних зменшений щонайменше на 3MB (з 4.78MB до ≤1.8MB загального розміру звуків).
- **SC-006**: Візуальний вигляд усіх 31 теми фону, усіх 31 скінів гравця та клавіатури ідентичний до оптимізації (перевірка скріншотами).
- **SC-007**: Жоден кадр не виконує більше 30 викликів `fillRect`/`strokeRect`/`arc` для частинкових систем одночасно.

## Assumptions

- Цільова роздільна здатність — 1366x768 (найпоширеніша для бюджетних ноутбуків). Оптимізації мають працювати і для Full HD, але пріоритет — HD.
- Браузер підтримує створення додаткових `HTMLCanvasElement` через `document.createElement('canvas')`.
- Аудіо-асети (`victory.wav`) можна конвертувати в MP3 без втрати якості відтворення у браузері.
- Усі фонові теми мають бути візуально ідентичні після переходу з shadowBlur на градієнти — підтверджується скріншотами.
- Демо-режим у меню також використовує оптимізований рендер (кешування фону, shadowBlur→gradient).
- FPS обмеження не впроваджується штучно — гра використовує рідний `requestAnimationFrame` із пропуском кадрів лише при великому dt.
- Частинкові системи фону (сніг, дощ, зірки) мають той самий ліміт, що й ігрові частинки (max 30), крім спеціалізованих систем як `_stars` для starfield (вони мають бути оптимізовані окремо через кешування).
