## Phase 3: User Story 1 — Базове відображення вікна стрибка при наближенні шипа (Priority: P1) 🎯 MVP

**Goal**: Індикатор з'являється на лінії землі від anchorX вперед на `this.okPx`, коли кубик живий і попереду є активний шип (state === "ahead"). Зникає, коли шип пройдено або кубик гине.

**Independent Test**: Запустити гру, дочекатися появи шипа — індикатор з'являється. Шип пройдено — індикатор зникає. Кубик гине — індикатор зникає.

### Implementation for User Story 1

- [x] T006 [US1] Реалізувати перевірку умов видимості в `renderHitWindow()`: `if (!this.player.alive || !this.nearestAheadSpike()) return;` у `js/engine.js`
- [x] T007 [US1] Додати малювання OK-зони (зовнішня, ціан) у `renderHitWindow()`: `fillRect` від `anchorX` до `anchorX + this.okPx`, висота 6 px з центром на `groundY`, колір `rgba(0, 246, 255, 0.15)`, `shadowBlur: 6` у `js/engine.js`
- [x] T008 [US1] Додати скидання `shadowBlur = 0` після малювання індикатора для уникнення артефактів у наступних елементах рендера у `js/engine.js`

**Checkpoint**: Індикатор (одна суцільна ціанова зона) з'являється при наближенні шипа і зникає після його проходження/смерті кубика. MVP досягнуто.

## Phase 4: User Story 2 — Розрізнення зони «ОК» та зони «Ідеально» (Priority: P2)

**Goal**: Всередині ціанової зони з'являється зелена зона (Perfect) з пульсуючою яскравістю та чіткою межею. Ціанова зона статична.

**Independent Test**: Під час гри візуально видно дві зони різного кольору з чіткою межею. Зелена зона пульсує, ціанова — ні.

### Implementation for User Story 2

- [x] T009 [US2] Додати малювання Perfect-зони (внутрішня, зелена) поверх OK-зони в `renderHitWindow()`: `fillRect` від `anchorX` до `anchorX + this.perfectPx`, висота 6 px з центром на `groundY`, базовий колір `rgba(57, 255, 136, 0.35)` у `js/engine.js`
- [x] T010 [US2] Реалізувати пульсацію яскравості Perfect-зони через `Math.sin(time * 6)`: обчислювати `alpha = 0.35 + 0.15 * (Math.sin(time * 6) + 1) / 2` (діапазон 0.35–0.50) та застосовувати до `rgba(57, 255, 136, alpha)` у `js/engine.js`
- [x] T011 [US2] Додати `shadowBlur: 10` та `shadowColor` відповідно до поточного alpha для Perfect-зони у `js/engine.js`
- [x] T012 [US2] Переконатися, що межа між OK-зоною та Perfect-зоною чітка (без градієнта) — OK-зона малюється першою, Perfect-зона другою поверх неї, обидві через `fillRect` без градієнтних стилів у `js/engine.js`

**Checkpoint**: Індикатор має дві візуально різні зони. Зелена пульсує; ціанова статична. Межа чітка.

## Phase 5: User Story 3 — Індикатор не заважає в періоди без шипів (Priority: P3)

**Goal**: Коли немає активних шипів попереду, індикатор відсутній. З'являється лише при state === "ahead".

**Independent Test**: Ділянки без шипів — індикатора немає. З'явився шип — індикатор з'явився. Шип пройдено — індикатор зник.

### Implementation for User Story 3

- [x] T013 [US3] Перевірити, що умова `this.nearestAheadSpike()` повертає `null` у проміжках між шипами та на початку рівня — валідація існуючої логіки; за потреби задокументувати крайові випадки у `js/engine.js`
- [x] T014 [US3] Перевірити візуально: на початку рівня (до першого шипа) індикатор відсутній; після проходження останнього шипа індикатор зникає; між серіями шипів індикатор коректно з'являється/зникає

**Checkpoint**: Індикатор візуально присутній лише коли є активний шип попереду. Жодного візуального шуму на лінії землі в решту часу.

## Implementation Strategy

### MVP First (User Story 1 Only)

1. Complete Phase 1: Setup — ознайомлення з кодом (T001–T003)
2. Complete Phase 2: Foundational — сигнатура методу + виклик (T004–T005)
3. Complete Phase 3: User Story 1 — базовий індикатор (T006–T008)
4. **STOP and VALIDATE**: Запустити гру, перевірити появу/зникнення індикатора
5. MVP досягнуто — індикатор працює

### Incremental Delivery

1. Setup + Foundational → метод викликається, але порожній
2. Add US1 → базова ціанова зона → Test → MVP!
3. Add US2 → зелена зона з пульсацією → Test → повноцінний індикатор
4. Add US3 → валідація крайових випадків → Test → production-ready
5. Polish → фінальна валідація за quickstart.md

### Single Developer Strategy

Весь обсяг змін — один файл (`js/engine.js`), один метод. Рекомендовано послідовне виконання Phases 1→2→3→4→5→6 через малий розмір задач.
