# Research: Ліги, гейміфіковані назви рівнів та адаптивний UX

**Feature**: 011-league-level-restructure
**Date**: 2026-07-19

## 1. Структура LEVELS_CONFIG

### Decision: Вкладений дворівневий масив «ліги → рівні»

Заміна поточного плаского масиву `LEVELS` (15 елементів) на структуру `LEVELS_CONFIG`:

```js
const LEVELS_CONFIG = [
  {
    id: 1,
    name: "Базова",
    levels: [
      { leagueId: 1, id: 1, name: "Перші кроки", letters: [...], ... },
      // ... 16 рівнів
    ]
  },
  {
    id: 2,
    name: "Середня",
    levels: [ /* 8 рівнів */ ]
  },
  // ... 5 ліг
];

// Зручний плоский доступ для швидкого пошуку:
const ALL_LEVELS = LEVELS_CONFIG.flatMap(l => l.levels);
```

**Rationale**: Зберігає зворотну сумісність із існуючими посиланнями на `LEVELS[n]` для ігрового циклу, дозволяючи плоский пошук через `ALL_LEVELS[idx]`. Ліги — суто UI-концепт, ігровий рушій працює з конкретним рівнем.

**Alternatives considered**:
- Окремий `LEAGUES` + `LEVELS` масиви з foreign key — надмірно, ускладнює пошук.
- Плаский масив із полем `leagueId` — незручно для групування в UI.

## 2. Схема збереження прогресу в localStorage

### Decision: Розширення існуючої схеми `dfp_save_v1` з новим полем `progress.completed`

Поточна схема:
```json
{
  "version": 1,
  "progress": {
    "unlocked": 1,
    "levels": { "1": { "bestPct": 0, "highScore": 0 }, ... }
  }
}
```

Нова схема:
```json
{
  "version": 1,
  "progress": {
    "unlocked": 1,
    "levels": { "1": { "bestPct": 0, "highScore": 0 }, ... }
  }
}
```

**Важливо**: Для прогресивного розблокування використовуємо **існуюче поле `unlocked`** (1-значний ID рівня, 1..31). Логіка `recordResult()` оновлюється: при 100% на рівні N у лізі L, встановлює `unlocked = наступний рівень у лізі`. Для переходу між лігами — після останнього рівня ліги N розблоковується перший рівень ліги N+1.

**Rationale**: Мінімальні зміни в SaveManager. Зворотна сумісність — старі збереження санітизуються до нового формату (рівні > 15 ігноруються).

**Alternatives considered**:
- Новий ключ `dfp_save_v2` — порушує зворотну сумісність, вимагає міграції.
- Окремий масив `unlockedPerLeague` — надмірно, поточна модель `unlocked` як глобальний ID простіша.

## 3. Механіка прогресивного розблокування

### Decision: Перший рівень кожної ліги доступний одразу. Всередині ліги — послідовне розблокування.

Логіка в `recordResult(levelId, pct, score)`:
1. При 100% — знайти поточний рівень у `ALL_LEVELS`
2. Якщо є наступний рівень у тій самій лізі — розблокувати його ID
3. Якщо це останній рівень ліги — розблокувати перший рівень наступної ліги (якщо є)
4. Бос-рівень (5-1, ID=31) — особливий: розблоковується після завершення всіх рівнів Майстрів (ліга 4)

**Rationale**: Простота. Перший рівень кожної ліги завжди `level.leagueId*1 - 1` (тобто `ALL_LEVELS.find(l => l.id === 1)` для ліги 1, `ALL_LEVELS.find(l => l.leagueId === 2 && l.id - його перший у лізі)`). Гравець може перемикатися між лігами без завершення попередньої.

**Alternatives considered**:
- Повне блокування ліг (спочатку вся Базова, потім уся Середня...) — обмежує свободу гравця, суперечить специфікації.

## 4. Ідентифікація рівнів (глобальний ID)

### Decision: Глобальна нумерація 1..31

| Ліга | Діапазон ID |
|------|-------------|
| 1. Базова (16 рівнів) | 1–16 |
| 2. Середня (8 рівнів) | 17–24 |
| 3. Складна (4 рівні) | 25–28 |
| 4. Майстер (2 рівні) | 29–30 |
| 5. Бос (1 рівень) | 31 |

Локальний номер у лізі = `level.id - firstLevelIdInLeague + 1`.
Формат відображення: `"L-N: Назва"` (наприклад, `"1-3: Ближче до країв"`).

**Rationale**: Зберігає просту цілочисельну адресацію. `Engine` отримує рівень за глобальним ID через `ALL_LEVELS.find(l => l.id === id)`.

## 5. UI: Вкладки ліг (DOM)

### Decision: Динамічне створення вкладок у `index.html` + стилізація в CSS

Додати контейнер `<div id="leagueTabs" class="league-tabs"></div>` над `<div id="levelGrid">`. Вкладки генеруються в `buildLevelCards()` циклом по `LEVELS_CONFIG`.

CSS: горизонтальний flex-ряд, активна вкладка з неоновим підсвічуванням, неактивна — тьмяна.

**Rationale**: Мінімальні зміни в DOM. Уся логіка в `main.js`.

**Alternatives considered**:
- Canvas-вкладки — суперечить існуючій архітектурі (вибір рівня на DOM, Canvas для гри).
- Статичний HTML — менш гнучко, складніше підтримувати.

## 6. HUD: Формат рядка «Ліга: X | N: Назва»

### Decision: Передача leagueInfo в Engine конструктор

Додати до `Engine` конструктора параметр `leagueInfo = { leagueName, levelNumber, levelName }`. Метод `renderProgressBar()` формує рядок:
```
Ліга: Базова | 1-3: Ближче до країв
```
відображається у верхній частині Canvas поверх прогресу або замість старого «Рівень X».

**Rationale**: Мінімальна зміна API Engine. Інформація про лігу передається з `main.js` при старті рівня.

## 7. Процедурні фони (31 варіант)

- `"dark_red_bezier"` — темно-червоний з хвилями Безьє (1-14)

**Alternatives considered**:

## 8. Оновлення посилань на рівні

### Decision: Пошук через `ALL_LEVELS` замість індексу

Усі місця, де використовується `LEVELS[idx]` або `level.id`, оновлюються:
- `startLevel(id)` → шукає рівень через `ALL_LEVELS`
- `buildLevelCards()` → ітерує `LEVELS_CONFIG` (ліги) → `league.levels` (рівні)
- `Engine` конструктор → отримує `level` об'єкт безпосередньо
- `handleVictory()` → використовує `currentLevelId` для пошуку
- `handleGameOver()` → аналогічно
- Демо-рушій → використовує `getLastPlayable()` (глобальний ID)

**Rationale**: Мінімальні зміни в логіці, лише пошук замість індексації.
