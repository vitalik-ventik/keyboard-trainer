# Research: Буферизація стрибка (003-air-jump-buffer)

**Date**: 2026-07-18 | **Plan**: [plan.md](./plan.md)

NEEDS CLARIFICATION відсутні. 5 рішень проти реального коду `js/engine.js`.

## R1. Вміст буфера: посилання на шип, а не літера

- **Decision**: `this.jumpBuffer` зберігає ПОСИЛАННЯ на об'єкт шипа
  (`{x, letter, state}`), який був найближчим попереду на момент натискання.
- **Rationale**: FR-040 вимагає гарантії «ніколи не стрибнути для вже
  пройденого шипа». Посилання дає точну перевірку `buffer.state === "ahead"`
  при приземленні. Літера такої гарантії не дає: наступний шип може мати ту
  саму літеру (траси генеруються з повторами в межах 2 однакових поспіль), і
  буфер «перескочив» би на інший шип, змінюючи семантику натискання.
- **Alternatives considered**:
  - Зберігати літеру — відхилено (колізія однакових літер сусідніх шипів).
  - Зберігати індекс шипа — еквівалентно посиланню, але потребує масиву за
    індексом; посилання простіше і вже узгоджене зі структурою `this.spikes`.

## R2. Умова буферизації в handleLetter

- **Decision**: Гілка ПЕРЕД перевіркою `inWindow`: якщо
  `correct && !this.player.onGround` → `this.jumpBuffer = spike; return;`
  (для обох режимів). Решта гілок `handleLetter` — без змін: неправильна
  літера в повітрі чи на землі → EASY ігнор / HARD вибух (FR-042); правильна
  на землі поза вікном → EASY ігнор / HARD вибух (FR-043).
- **Rationale**: Мінімальна точкова зміна: буферизуються лише правильні
  натискання в повітрі (FR-037); `return` до HARD-гілки вибуху прибирає
  стару кару (SC-014). Перезапис (FR-039) відбувається природно —
  присвоєння затирає попереднє значення.
- **Alternatives considered**:
  - Буферизувати й помилкові літери «на прощення» — відхилено: суперечить
    FR-042 і розмиває HARD.

## R3. Точка спрацювання: перехід onGround false→true в update()

- **Decision**: У гілці вертикальної фізики `update()`, одразу після
  `player.y <= 0 → onGround = true`, викликається приватна перевірка буфера:
  1. `buffer === null` → нічого;
  2. `buffer.state !== "ahead"` → `buffer = null` (шип пройдено/збито);
  3. `gap = buffer.x - player.x`; якщо `0 < gap ≤ okPx` → виконати стрибок
     (позначити `cleared`, нарахувати очки з бонусом за фактичним gap,
     `jump()` → звук через наявний колбек `onJump`), `buffer = null`;
  4. інакше (gap ≤ 0 або gap > okPx) → `buffer = null` тихо (Assumption зі
     spec: недосяжний шип → скасування без покарання).
- **Rationale**: Це єдине місце переходу `false → true` у коді (фізика
  приземлення), що виконується ДО перевірки «HARD-пропуск» і колізій у тому ж
  кадрі — автостибок встигає врятувати кубик (SC-015). Кейс «переписати
  gap > okPx і чекати» відхилено: перетворив би буфер на автогру.
- **Alternatives considered**:
  - Перевірка буфера на початку кожного `update()` — відхилено: спрацює й
    без факту приземлення (кубик стоїть на землі), змінюючи семантику
    «натиснув у повітрі → стрибок після приземлення».

## R4. Точки скасування буфера

- **Decision**: `jumpBuffer = null` встановлюється у: (а) `explode()` —
  покриває і фізичне зіткнення, і HARD-вибухи, включно з «пропуском
  таймінгу» (у HARD пропуск = вибух у поточній реалізації); (б) перевірці
  приземлення (R3, пункти 2 і 4); (в) `reset()` — ініціалізація на старті
  та рестарті забігу (FR-041, вимога користувача №4); (г) неявно — при
  перезаписі новим натисканням (R2).
- **Rationale**: Повне покриття трьох випадків FR-040 + життєвий цикл
  забігу. В EASY «пропуск» шипа завершується або фізичним зіткненням
  (→ explode → (а)), або перельотом (spike стає `cleared` → (б) пункт 2).
- **Alternatives considered**:
  - Таймер життя буфера (мс) — відхилено: політ кубика короткий (~0.5 с),
    життєвий цикл і так обмежений найближчим приземленням.

## R5. Відсутність змін публічного API та інших модулів

- **Decision**: Жодних нових публічних методів/полів у контракті Engine;
  `jumpBuffer` — внутрішнє поле. `main.js`, `keyboard.js`, `assets.js`,
  `index.html`, `css/style.css` не змінюються. `getTargetLetter()` й далі
  повертає літеру найближчого шипа зі станом `ahead` — після забуференого
  автостибка підсвітка природно переходить на наступний шип.
- **Rationale**: Вимога користувача №1; контракти фічі 001 лишаються
  чинними; демо-режим (`demoMode`) не потребує буфера — `handleLetter`
  виходить раніше, автогра стрибає власною логікою.
- **Alternatives considered**:
  - Експонувати стан буфера для підсвітки клавіатури (клавіша «в черзі») —
    відхилено: розширює scope на keyboard.js/main.js всупереч вимозі №1;
    можлива окрема майбутня фіча.
