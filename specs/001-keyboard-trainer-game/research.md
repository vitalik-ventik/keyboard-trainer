# Research: Дитячий клавіатурний тренажер (001-keyboard-trainer-game)

**Date**: 2026-07-18 | **Plan**: [plan.md](./plan.md)

У Technical Context немає маркерів NEEDS CLARIFICATION — стек жорстко заданий
Конституцією. Нижче зафіксовано ключові технічні рішення, які закривають
ризикові місця реалізації.

## R1. Спосіб запуску: ES6 Modules і протокол file://

- **Decision**: Застосунок запускається через локальний статичний сервер
  (`python -m http.server`, VS Code Live Server або будь-який аналог) на
  `http://localhost`. `index.html` підключає єдиний
  `<script type="module" src="js/main.js">`.
- **Rationale**: ES6 `import` та `fetch()` аудіофайлів блокуються CORS-політикою
  браузера на протоколі `file://`. Локальний сервер — нуль залежностей у коді
  проєкту (сервер — інструмент запуску, не частина застосунку), офлайн-вимога
  не порушується (localhost не потребує Інтернету).
- **Alternatives considered**:
  - Один великий JS-файл без модулів (працює з file://) — відхилено: порушує
    Принцип II (модульна архітектура).
  - Інлайн-аудіо в base64 — відхилено: порушує вимогу «локальні файли
    sounds/*.wav, music/*.mp3» і роздуває код.

## R2. Аудіо-архітектура: Web Audio API

- **Decision**: `assets.js` вантажить усі 8 файлів через
  `fetch → arrayBuffer → AudioContext.decodeAudioData`. Ефекти відтворюються
  через `AudioBufferSourceNode` (одноразові, можуть накладатися), музика — теж
  через `AudioBufferSourceNode` з `loop = true` та власним `GainNode`
  (окремі рівні гучності музики/ефектів). Кожне завантаження і відтворення —
  у try/catch; відсутній файл позначається `null` і тихо пропускається.
- **Rationale**: Ритм-гра вимагає мінімальної затримки відтворення ефектів
  (<10 мс) і накладання звуків (кілька стрибків підряд) — `HTMLAudioElement`
  дає затримки і обмеження одночасності. Розмір треків (0.2–2 МБ) дозволяє
  повне декодування в пам'ять.
- **Autoplay policy**: `AudioContext` створюється у стані `suspended`;
  розблокування — `context.resume()` на перший `pointerdown`/`keydown`
  (закриває edge case «музика не грає до першої взаємодії», без помилок у
  консолі).
- **Alternatives considered**:
  - `HTMLAudioElement` + `cloneNode()` для ефектів — відхилено: непередбачувана
    затримка, глюки накладання в Edge.
  - Потокова музика через `<audio>` + Web Audio для ефектів — відхилено:
    два аудіошляхи ускладнюють перемикання треків за станом.

## R3. Незалежність від системної розкладки: event.code

- **Decision**: `keyboard.js` зіставляє фізичні коди клавіш
  (`event.code`: `KeyF`, `KeyA`, `Semicolon`, `BracketLeft`, …) з українськими
  літерами ЙЦУКЕН через статичну мапу. `event.key` не використовується для
  ігрової логіки.
- **Rationale**: `event.code` не залежить від активної системної розкладки та
  регістру — вимоги FR-022 (зчитування укр. літер незалежно від розкладки і
  регістру) виконується автоматично. Дитині не потрібно перемикати мову в
  Windows.
- **Мапа**: 33 літери на трьох рядах фізичної QWERTY-клавіатури:
  ряд 1 — `Й Ц У К Е Н Г Ш Щ З Х Ї` (KeyQ…BracketRight),
  ряд 2 — `Ф І В А П Р О Л Д Ж Є` (KeyA…Quote), `Ґ` — Backslash,
  ряд 3 — `Я Ч С М И Т Ь Б Ю` (KeyZ…Period).
- **preventDefault**: викликається для Space, стрілок, Tab і всіх ігрових
  клавіш у станах, де активна гра/меню (FR-023).
- **Alternatives considered**:
  - `event.key.toLowerCase()` — відхилено: ламається на латинській розкладці
    (гравець мусив би перемикати мову ОС).

## R4. Ігровий цикл: rAF з обмеженим delta

- **Decision**: Один `requestAnimationFrame`-цикл у `main.js`;
  `dt = min((now - last) / 1000, 0.05)` секунд (кламп 50 мс). Уся фізика —
  функції від `dt`. Додатково: на `visibilitychange` (вкладка прихована) гра в
  стані PLAYING ставиться на паузу вибухобезпечно (скидання `last`).
- **Rationale**: Кламп delta усуває «телепорт у шип» після згортання вікна
  (edge case зі spec, FR-027); варіативний timestep із клампом простіший за
  fixed-step акумулятор і достатній для аркадної точності.
- **Alternatives considered**:
  - Fixed timestep + інтерполяція — відхилено: зайва складність для 2D-аркади
    без мережі.

## R5. Збереження: схема localStorage з fallback

- **Decision**: Один ключ `dfp_save_v1` із JSON-об'єктом (див.
  [contracts/storage-schema.md](./contracts/storage-schema.md)). Обгортка
  SaveManager: `load()`/`save()` у try/catch; при недоступності storage —
  робота з об'єктом у пам'яті (прогрес живе до закриття вкладки), без жодних
  помилок у консолі. Рекорди пишуться тільки якщо кращі за збережені
  (FR-015). Пошкоджений JSON → скидання до дефолту.
- **Alternatives considered**:
  - Окремі ключі на кожне поле — відхилено: більше точок відмови, неатомарно.
  - IndexedDB — відхилено: надлишково для <1 КБ даних.

## R6. Фіксовані траси рівнів

- **Decision**: Кожен рівень описується декларативною константою в `engine.js`:
  швидкість, набір літер, масив шипів `{dist, letter}` — згенерований
  детермінованим PRNG (mulberry32 з фіксованим сідом рівня) один раз під час
  ініціалізації рівня. Однаковий сід → однакова траса при кожному запуску
  (FR-016). Рівень 6: групи по 2–4 шипи з ритмічними інтервалами.
- **Rationale**: Детермінована генерація дає «фіксовану трасу» без ручного
  набивання 50+ координат і гарантує рівний розподіл літер групи.
- **Alternatives considered**:
  - Повністю ручні масиви трас — відхилено: багатослівно, важко балансувати.
  - `Math.random()` щоразу — відхилено: порушує вимогу фіксованості трас.

## R7. Canvas: адаптивність та чіткість

- **Decision**: Canvas розтягується на все вікно (CSS 100vw/100vh); внутрішній
  розмір = `clientSize × devicePixelRatio` з `ctx.setTransform(dpr,0,0,dpr,0,0)`.
  Уся геометрія рахується від логічних розмірів `W×H`, перерахунок на
  `resize`. Ігрові пропорції — відносні (земля на 72% висоти, клавіатура —
  нижні 26%).
- **Rationale**: Чіткий неон без «мила» на дисплеях Windows 11 із масштабом
  125–150%; адаптивність (FR-029) без медіазапитів для ігрового поля.
- **Alternatives considered**:
  - Фіксований 1280×720 з letterbox — відхилено: гірша адаптивність, смуги.

## R8. DOM vs Canvas для UI

- **Decision**: Ігрове поле, кубик, шипи, частинки, trail, фони, візуальна
  клавіатура, смуга прогресу, демо-заставка — виключно Canvas. DOM (стилізація
  у `css/style.css`) — лише для: екрана завантаження, кнопок головного меню,
  модального вікна налаштувань, сітки карток вибору рівня, оверлеїв
  поразки/перемоги.
- **Rationale**: Конституція забороняє DOM для ігрових об'єктів, але роль
  UI/UX Engineer прямо передбачає «стилізацію DOM-елементів» для меню і
  модалок; DOM-кнопки дають доступність і простіші hit-тести, ніж Canvas-кнопки.
- **Alternatives considered**:
  - 100% Canvas включно з меню — відхилено: ручні hit-тести, гірша типографіка
    модалок, більше коду без виграшу.
