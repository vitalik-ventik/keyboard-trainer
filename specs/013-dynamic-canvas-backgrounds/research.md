# Research: Динамічні багатошарові Canvas-фони

**Feature**: 013-dynamic-canvas-backgrounds
**Date**: 2026-07-19

## 1. Архітектурне рішення: винесення фонів в окремий модуль

### Decision
Створити `js/backgrounds.js` як окремий ES6-модуль, що експортує об'єкт `BackgroundRenderer` із методами для кожного типу фону.

### Rationale
- **Принцип II** конституції вимагає модульної архітектури з єдиною відповідальністю
- Поточний `engine.js` (2108 рядків) містить ~35 методів рендерингу фону, що становить ~40% файлу
- Винесення дозволяє тестувати та модифікувати фони незалежно від ігрової логіки
- Спрощує майбутнє розширення (додавання нових типів фонів без зміни engine.js)

### Alternatives Considered
- **Залишити все в engine.js**: Відхилено — порушує Принцип II, ускладнює підтримку
- **Окремий файл на кожен тип фону (10 файлів)**: Відхилено — надмірна фрагментація без вигоди
- **Клас із реєстрацією фонів (plugin-патерн)**: Відхилено — надмірна складність для 10 типів

## 2. Canvas 2D Performance Optimizations

### Decision
Використовувати наступні техніки для підтримання стабільних 60 FPS:

1. **Умовне застосування `shadowBlur`**: Вмикати лише для основних ліній сітки та контурів перешкод. Для фонових елементів використовувати напівпрозорі накладення замість shadowBlur (дешевший рендер).

2. **Обмеження кількості draw-викликів**: Кожен тип фону має виконувати не більше 30-40 `stroke()`/`fill()` викликів за кадр. Для масових елементів використовувати `beginPath()` з одним `stroke()`.

3. **Попереднє обчислення констант**: Усі значення, що залежать лише від розміру Canvas (W, H), обчислюються один раз при ініціалізації та при resize.

4. **Уникнення створення об'єктів у циклі рендеру**: Використовувати пули об'єктів для часток, крапель, стовпчиків. Перевикористовувати існуючі об'єкти замість створення нових.

5. **Оптимізація тексту**: Для матричного потоку використовувати `ctx.fillText()` з фіксованим шрифтом; уникати частих змін `ctx.font`.

### Rationale
Canvas 2D shadowBlur є одним із найдорожчих ефектів (може сповільнити рендер у 3-5 разів). Обмеження його застосування критичне для стабільних 60 FPS. Попереднє обчислення та пули об'єктів зменшують навантаження на GC.

### Alternatives Considered
- **Off-screen canvas для попереднього рендеру**: Відхилено — додає складність, а фон змінюється щокадру; off-screen canvas вигідний лише для статичних або рідко оновлюваних шарів.
- **WebGL для фонів**: Відхилено — порушує конституційне обмеження (лише Canvas 2D).

## 3. Математичні моделі для 10 типів фонів

### 3.1 Кібер-Сітка (рівні 1-1, 1-5)
**Модель**: Перспективна проекція з точкою сходу на горизонті.
- Горизонт: `horizonY = H * 0.35`
- Точка сходу: `(W/2, horizonY)`
- Вертикальні лінії: `x = W/2 + (i - cols/2) * spacing`, де `spacing` збільшується з відстанню від горизонту
- Горизонтальні лінії: `y = horizonY + offset + j * dy`, де `dy` зменшується при наближенні до горизонту (ефект перспективи)
- Рух: `offset += speed * dt`, циклічний з reset
- Glow: `shadowBlur = 12`, колір сітки = акцентний колір рівня

### 3.2 Паралакс Міста (рівні 1-2, 2-1)
**Модель**: 3 шари з різною швидкістю прокрутки.
- `layerSpeeds = [0.3, 0.6, 1.0] * baseSpeed` (задній, середній, передній)
- Кожен шар: масив прямокутників різної висоти (30-60% від H) з випадковими інтервалами
- Вікна: `фазовий зсув + Math.sin(time * randFreq + randPhase) > 0.5` → колір вікна
- Кольори: задній #1a0a2e, середній #2d1b4e, передній #3d2b5e, вікна #ffcc00/#ff6600

### 3.3 Гіперпросторовий Тунель (рівні 1-11, 3-3)
**Модель**: Масштабування прямокутників від центру з нелінійним прискоренням.
- `scale = 1 + Math.pow(phase, 2.5) * 3` (нелінійне прискорення через степеневу функцію)
- `phase = (time * speed) % 1` (циклічний)
- Колір: HSL-цикл `hsl((time * 60 + i * 15) % 360, 80%, 50%)`
- 8-12 прямокутників, ширина/висота пропорційні меншій стороні Canvas

### 3.4 Синусоїдальні Хвилі (рівні 1-14, 4-2)
**Модель**: 4 накладені синусоїди з різними параметрами.
- Частоти: `[0.8, 1.3, 2.1, 3.5]`
- Амплітуди: `[H*0.06, H*0.08, H*0.05, H*0.07]`
- Фазові зсуви: `time * [0.5, 0.7, 1.0, 1.3]`
- Кольори: неонові (magenta, cyan, yellow, lime)
- `globalAlpha = [0.25, 0.3, 0.2, 0.25]`
- Рендер через `ctx.bezierCurveTo()` або `ctx.quadraticCurveTo()` для гладкості

### 3.5 Неоновий Дощ (рівні 1-12, 2-3, 3-4)
**Модель**: Масив об'єктів-крапель (макс. 80).
- Кожна крапля: `{x, y, speed, length, color, life}`
- Падіння під кутом 45°: `dx = speed * 0.707, dy = speed * 0.707`
- Рендер: градієнтна лінія від `#00ffff` (яскравий) до transparent
- Іскри при досягненні низу: 3 нових об'єкти часток з коротким життям (0.2-0.3s)
- Нові краплі генеруються з випадковим інтервалом у верхній частині екрана

### 3.6 Матричний Потік (рівень 2-4)
**Модель**: Масив стовпчиків (макс. 40).
- Кожен стовпчик: `{x, y, speed, chars[], headAlpha}`
- Символи: випадкові "0" або "1" (оновлюються випадково при русі)
- Колір: `#00ccff` (кібер-синій), з білим для головного символу
- Ефект хвоста: кожен символ має alpha, що зменшується від голови до хвоста
- Шрифт: моноширинний, 14-18px

### 3.7 Еквалайзер (рівень 3-1)
**Модель**: N стовпчиків (N = 16-24) по нижньому краю.
- Ширина стовпчика: `W / N - gap`
- Висота `h[i] = H * (0.15 + 0.25 * (Math.sin(t*f1 + i*p1) + 0.5) * (Math.sin(t*f2 + i*p2) + 0.5) * (Math.cos(t*f3 + i*p3) + 0.5))`
- Параметри: `f1=2.7, f2=1.3, f3=4.1, p1=0.4, p2=0.7, p3=1.1`
- Колір: градієнт від яскравого (#ff00ff) до темного (#330044)
- Glow: `shadowBlur = 8` на стовпчиках

### 3.8 Зоряне Поле (нові рівні з FR-009: 1-3, 1-15, 3-5)
**Модель**: N зірок (N = 100-150) з 3 шарами глибини.
- Кожна зірка: `{x, y, z, brightness, twinklePhase}`
- Шари: `z = 0.3, 0.6, 1.0` (швидкість руху обернено пропорційна z)
- Рух: зірки рухаються вліво (`x -= speed * z`), з'являються справа при виході за ліву межу
- Мерехтіння: alpha модулюється `Math.sin(time * twinkleFreq + twinklePhase)`
- Колір: білий/блакитний із варіацією

### 3.9 Пульсуюча Енергетична Решітка (нові рівні з FR-009: 1-4, 2-5, 3-2)
**Модель**: Концентричні кола + діагональні промені від центру.
- `rings = 5-7` кілець, що пульсують з різною фазою
- Радіус `r[i] = r_base + amp * Math.sin(time * freq + i * phaseShift)`
- Промені: 8-12 ліній від центру до країв
- Колір: циклічний HSL зі зміщенням від часу
- Glow: `shadowBlur = 10` на кільцях

### 3.10 Геометричний Ландшафт (нові рівні з FR-009: 1-6, 1-7, 2-2, 4-1)
**Модель**: Процедурно згенерований полігональний рельєф.
- Генерація: масив точок `{x, y}` через фіксований інтервал по ширині
- Висота кожної точки: `y = groundY - H * 0.15 - H * 0.1 * perlin-like noise(x * 0.003 + time * 0.2)`
- Шум: спрощений через `Math.sin(x*f1 + t)*Math.cos(x*f2 - t)*0.5 + 0.5`
- Зсув: увесь ландшафт зміщується вліво зі швидкістю гри
- Колір: темний силует із неоновою обводкою (#00ff88)
- Два шари: задній (темніший, повільніший) + передній (яскравіший, швидший)

## 4. Система часток при руйнуванні

### Decision
Розширити існуючу систему часток в engine.js (клас Engine, метод `explode()`).

### Параметри
- Кількість: 15-20 часток (випадково)
- Початкова швидкість: 80-200 px/s у випадковому напрямку (кут 0-360°)
- Гравітація: 300-500 px/s² (тягне вниз)
- Час життя: 0.4-0.6 с (випадково)
- Колір: випадковий з палітри `["#00f6ff", "#ff2ea6", "#ffe14d", "#00ff88"]`
- Рендер: маленькі квадрати (2-4px) з `globalAlpha = life / maxLife`

### Optimization
- Максимум 50 одночасних часток (FR-013)
- При перевищенні: найстаріші (з найменшим `life`) видаляються миттєво
- Масив часток очищається при кожному новому запуску рівня

## 5. Glow-оптимізація

### Decision
Застосовувати `ctx.shadowBlur` лише для:
1. Основних ліній сітки (не для всіх 100+ ліній)
2. Контурів перешкод (spike/saw outlines)
3. Горизонту (для сітки)

### Параметри
- `shadowBlur = 15` для основних структурних ліній
- `shadowColor = currentStrokeColor`
- Для масових елементів (частки, символи матриці, краплі) — НЕ використовувати shadowBlur
- Альтернатива: подвійне малювання — спочатку товстіша напівпрозора лінія, потім тонка яскрава
