# Research: Реєстр скінів гравця (31 рівень)

**Дата**: 2026-07-19

## 1. Структура поля `skin` у рівнях

### Decision
Кожен об'єкт підрівня в `LEVELS_CONFIG` отримує поле `skin`:
```javascript
skin: { id: "skin_1_1", name: "Стандартний Неон", renderType: "neon_base" }
```

### Rationale
- `id` — унікальний рядковий ключ для відстеження розблокувань (напр. `"skin_1_1"`), що відповідає рівню `1-1`
- `name` — українська назва для сповіщення про розблокування
- `renderType` — ключ диспетчеризації, який використовується для вибору функції рендерингу в `renderPlayer()`
- Мінімальна структура — легко серіалізується та не перевантажує конфігурацію

### Alternatives considered
- **Зберігати всі параметри рендерингу в `skin` (колір, розміри тощо)**: Відхилено — рендеринг скінів надто різноманітний (від простих кольорів до спіралей і корон); єдиний уніфікований набір параметрів неможливий
- **Зберігати `skin` як простий рядок (лише renderType)**: Відхилено — потрібні `id` для localStorage і `name` для сповіщення

## 2. Диспетчеризація рендерингу в `renderPlayer()`

### Decision
Використовувати об'єкт-словник `SKIN_RENDERERS` поза межами методу `renderPlayer()`:
```javascript
const SKIN_RENDERERS = {
    neon_base: function(ctx, size, time) { /* рендер */ },
    cyber_eye: function(ctx, size, time) { /* рендер */ },
    // ... 31 записів
};
```
У `renderPlayer()`:
```javascript
const renderFn = SKIN_RENDERERS[this.level.skin.renderType];
if (renderFn) {
    renderFn(ctx, CUBE_SIZE, this.currentTime);
} else {
    SKIN_RENDERERS.neon_base(ctx, CUBE_SIZE, this.currentTime);
}
```

### Rationale
- Словник швидший за `switch` із 31 гілкою (O(1) проти O(n))
- Кожна функція ізольована — легко тестувати окремо
- `this.currentTime` передається для динамічних скінів («Плазма», «Подвійна Рамка»); джерело — `performance.now()` або лічильник кадрів
- Функції отримують `ctx` у стані після `translate`/`rotate` (центр кубика вже в (0,0))

### Alternatives considered
- **Методи в об'єкті Player**: Відхилено — Player є простим об'єктом (не клас), додавання 31 методу роздуває об'єкт
- **Окремий файл `skins.js`**: Відхилено — створює додатковий модуль, що ускладнює структуру без потреби (Принцип II: модулі з єдиною відповідальністю — усі скіни є частиною Engine)

## 3. Механізм збереження розблокувань скінів

### Decision
Додати поле `unlockedSkins: []` (масив рядкових ідентифікаторів) до `saveData.progress` та ключі для кожного рівня в `saveData.progress.levels[id]`:
- У `defaultSaveData()`: `unlockedSkins: []` (порожній масив)
- У `save.recordResult()`: при 100% проходженні додавати `level.skin.id` до масиву, якщо ще немає
- У `sanitizeSaveData()`: валідувати `unlockedSkins` як масив рядків
- У `save.load()`: забезпечити зворотну сумісність (якщо поле відсутнє — створити порожній масив)

### Rationale
- Використання існуючого ключа `"dfp_save_v1"` уникає міграції даних
- Перевірка наявності скіна перед додаванням запобігає дублюванню при повторному проходженні
- Масив рядків мінімальний за розміром (~31 елемент × ~10 символів = ~310 байт)

### Alternatives considered
- **Новий ключ localStorage**: Відхилено — ускладнює управління даними та потребує синхронізації двох ключів
- **Бітова маска (31 біт)**: Відхилено — менш читабельна, складніша для налагодження

## 4. Сповіщення про розблокування на екрані VICTORY

### Decision
Модифікувати `handleVictory()` у `js/main.js`:
- Отримати `currentLevel.skin` з `ALL_LEVELS`
- Викликати `save.isSkinNewlyUnlocked(skin.id)` або перевірити всередині
- Якщо скін новий — встановити текст у `victoryUnlockEl`: `«Розблоковано новий скін: [Назва]!»`
- Якщо рівень уже проходили — не показувати

### Rationale
- `handleVictory()` уже має логіку для `victoryUnlockEl` (показує наступний рівень); розширення мінімальне
- Перевірка «першого проходження» вже є через `resultRecorded`; потрібна додаткова перевірка наявності скіна в `unlockedSkins`

### Alternatives considered
- **Окремий модальний діалог**: Відхилено — надлишково; поточний `victoryUnlockEl` достатній
- **Canvas-повідомлення**: Відхилено — ускладнює реалізацію; DOM-елемент простіше і вже існує

## 5. Підхід до реалізації 31 функції рендерингу

### Decision
Кожна функція рендерингу має сигнатуру:
```javascript
function renderSkin(ctx, size, time)
```
де:
- `ctx` — Canvas 2D контекст після `translate(0,0)` та `rotate(rotation)` (центр кубика в (0,0))
- `size` — `CUBE_SIZE` (42 пікселі)
- `time` — `performance.now()` для динамічних ефектів (ігнорується статичними скінами)

Координати малювання відраховуються від центру: `-size/2` до `+size/2`.

Функції організовані групами за лігами (коментарі-роздільники):
- `// === ГРУПА 1: БАЗОВА ЛІГА ===`
- `// === ГРУПА 2: СЕРЕДНЯ ЛІГА ===`
- `// === ГРУПА 3: СКЛАДНА ЛІГА ===`
- `// === ГРУПА 4: ЛІГА МАЙСТРІВ ===`
- `// === ГРУПА 5: ЛІГА БОСА ===`

### Rationale
- Уніфікована сигнатура спрощує диспетчеризацію
- Роздільники-коментарі полегшують навігацію у файлі (~800 рядків коду)
- Координати відносно центру відповідають тому, як уже працює `renderPlayer()` (рядки 939-941)

## 6. Спеціальні випадки рендерингу

### 6.1 Скін «Матричний Піксель» (1-6): випадкові квадрати

**Decision**: Генерувати 3 випадкові позиції при першому рендері, зберігати в замиканні або як властивість рівня для стабільності.

**Rationale**: Щокадрова регенерація створює мерехтіння, що не відповідає опису «імітація коду». Фіксовані позиції на весь рівень виглядають стабільніше.

### 6.2 Скін «Метеор» (3-4): шлейф із копій

**Decision**: Додати `meteorTrail: []` (масив із 2 елементів `{x, y, alpha}`) до об'єкта `player`. Оновлювати в `update()` (engine.js:588-699): при кожному кадрі зсувати попередні позиції, додавати нову. У `SKIN_RENDERERS.meteor` малювати 2 напівпрозорі копії з `globalAlpha`.

**Rationale**: Окремий масив не заважає існуючому `player.trail` (який використовується для базового шлейфу). Зберігання в `player` дозволяє очищати при `reset()`.

### 6.3 Динамічні скіни: «Плазма» (3-1) та «Подвійна Рамка» (1-9)

**Decision**: Використовувати `time` параметр із `performance.now()`. Для «Плазма» — радіус радіального градієнта = `size/2 + sin(time * 0.003) * 8`. Для «Подвійна Рамка» — відступ рамки = `2 + sin(time * 0.005) * 2`.

**Rationale**: `performance.now()` забезпечує плавну анімацію незалежно від FPS. Множники підібрані для візуально приємної швидкості пульсації.

## 7. Зворотна сумісність даних

### Decision
При завантаженні збережень, де `unlockedSkins` відсутнє (гравці з попередніх версій), додавати порожній масив `[]`. Скін для поточного рівня завжди рендериться незалежно від стану розблокування (гравець бачить скін рівня, який проходить).

### Rationale
- Уникнення помилок при читанні старих даних
- Поточний скін рівня рендериться завжди (не потребує попереднього розблокування) — це візуальна винагорода за проходження
