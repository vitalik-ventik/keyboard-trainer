# AI Handoff & Context (AGENTS.md)

Цей документ визначає ролі, архітектурні обмеження та правила поведінки для штучного інтелекту (LLM / AI-агентів), які беруть участь у розробці та модифікації проекту **KeyboardTrainer** (Клавіатурний тренажер у стилі Geometry Dash).

---

## 1. Ролі ШІ-Агентів (AI Roles)

Будь-яка модель, яка працює з цим репозиторієм, повинна приймати одну з наступних ролей залежно від запиту користувача:

### [Lead Architect]
- **Обов'язки:** Контроль цілісності архітектури модульного веб-застосунку.
- **Правило:** Заборонено пропонувати перехід на Node.js, Webpack, React, Vue або інші фреймворки. Проект має залишатися у межах **чистого HTML5 (Canvas) + Vanilla JS (ES6 Modules)** із розділенням на логічні файли та папки.

### [UI/UX Engineer]
- **Обов'язки:** Рендеринг інтерфейсу меню, модальних вікон та візуальної клавіатури на Canvas, а також стилізація DOM-елементів.
- **Правило:** Дотримуватися стилістики темного неонового кіберпанку (Geometry Dash). Враховувати адаптивність під роздільну здатність екрана Windows 11.

### [Core Developer]
- **Обов'язки:** Написання ігрового циклу (`requestAnimationFrame`), обробка колізій, менеджер станів (State Manager) та обробка введення (`keydown`).
- **Правило:** Писати чистий, оптимізований ES6+ код без використання сторонніх npm-пакетів.

---

## 2. Технічний стек та Обмеження (Tech Stack & Constraints)

ШІ-агент повинен суворо дотримуватися цих обмежень. Порушення будь-якого з пунктів вважається критичною помилкою:

- **Мова інтерфейсу та коментарів:** Українська.
- **Формат проекту:** Модульний Vanilla JS застосунок. Код обов'язково розділяється на `index.html`, `css/style.css` та окремі JavaScript-файли в папці `js/` (`main.js`, `assets.js`, `engine.js`, `keyboard.js`), пов'язані через ES6-імпорти або правильний порядок підключення.
- **Графічний рушій:** HTML5 Canvas (2D Context). Жодного DOM-маніпулювання для ігрових об'єктів (кубик, шипи).
- **Звук та Музика:** Асинхронне завантаження зовнішніх локальних файлів (`sounds/*.wav` та `music/*.mp3`). **Категорично заборонено** використовувати аудіо-посилання на зовнішні CDN чи хостинги.
- **Збереження прогресу:** Виключно локальний `localStorage`.

---

## 3. Специфікація Логіки (Core Gameplay Spec)

Під час генерації або оновлення коду ШІ повинен враховувати:

1. **Механіка введення:** Тільки українська розкладка ЙЦУКЕН. Перехоплення події `keydown` має скасовувати дефолтну поведінку Windows (заборона скролу сторінки через Пробіл або Стрілки).
2. **Візуальна клавіатура:** Рендериться виключно на Canvas. Відображає лише блоки з українськими літерами розкладки ЙЦУКЕН. Службові клавіші (Shift, Enter, CapsLock) відображати чи прораховувати **не потрібно**.
3. **Режими складності:**
   - `EASY`: Ігнорування помилкових натискань завчасно. Кубик вибухає лише при фізичному зіткненні з шипом.
   - `HARD`: Будь-яка помилка, пропуск таймінгу чи натискання не тієї літери — миттєвий вибух.
4. **Процедурні бекграунди:** Код повинен рендерити паралакс-ефект на тлі Canvas (сітки, геометричні гори, пульсуючі в такт гри еквалайзери для Бос-рівня), а не посилатися на статичні зображення.

---

## 4. Правила генерації коду для AI (Code Generation Rules)

1. **Жодних скорочень:** Заборонено використовувати коментарі на кшталт `// тут додайте логіку`, `// решта коду без змін`. Код кожної функції чи класу в межах таски має бути написаний повністю від початку до кінця.
2. **Безпека типів:** Перед використанням `localStorage` або завантаженням аудіо завжди реалізовувати блоки `try/catch` для запобігання фатальних помилок у консолі Windows 11.
3. **Стан гри:** Чітко розділяти ігрові стани (`MENU`, `SETTINGS`, `LEVEL_SELECT`, `PLAYING`, `GAMEOVER`, `VICTORY`). Музичні треки мають перемикатися відповідно до активного стану.

# Custom Workflows

## Workflow: SpecKit Full Cycle
When I use the keyword `@autospec [description]`, you must execute the 
following chain of agent skills in this strict order:
1. Call the `/speckit.specify` plugin with the provided description.
2. Analyze the generated `SPECIFICATION.md` file and call `/speckit.plan`.
3. Generate tasks based on `PLAN.md` by calling `/speckit.tasks`.
4. Hand over control to the Build Agent by calling `/speckit.implement` for the first task listed in `TASKS.md`.
